# E2EE v3 – Group Chat Decryption Bug on New Member Join

## Overview

E2EE is functioning correctly across private and group chats. However, a critical issue occurs when a **new member joins a group chat**, affecting message visibility for existing members.

---

## Problem Summary

### Current Behavior

#### Before New Member Joins

* All existing members:

  * Can see and decrypt all past messages
  * Chat works correctly

---

#### After New Member Joins

##### New Member

* Cannot see past messages
* ✅ This is correct behavior (expected in E2EE)

---

##### Existing Members (BUG)

* Previously visible messages:

  * Become **blank** or stuck in loading state
* Decryption fails for old messages

---

## Key Issue

> When a new member joins, **existing members lose access to past messages**.

This should **never happen**.

---

## Root Cause (Actual)

### ❗ Group Key Rotation Without Proper Handling

When a new member joins, your system likely:

* Generates a **new group encryption key**
* Applies it to the conversation

However:

### What Goes Wrong

* Old messages were encrypted with **OLD key**
* New key is now used globally
* Existing members:

  * Try to decrypt old messages using **NEW key**
  * ❌ Decryption fails → messages become blank

---

## Why Your Hypothesis is Close (But Not Exact)

You said:

> “Some algorithm hides previous messages”

👉 What’s actually happening:

* Messages are **not hidden**
* They are **failing to decrypt**
* UI shows blank because decryption returns nothing

---

## Correct E2EE Behavior for Groups

### Rule:

> Each message must be decrypted using the **same key it was encrypted with**

---

## Suggested Fixes

### Fix 1: Key Versioning (Recommended)

Each message should store which key was used:

```ts
{
  content: "...",
  keyId: "group-key-v1"
}
```

Then:

```ts
const key = getKeyById(message.keyId);
decrypt(message, key);
```

---

### Fix 2: Do NOT Reuse New Key for Old Messages

When a new member joins:

* Generate new key ✅
* Use it ONLY for:

  * future messages

❌ Do NOT apply it to old messages

---

### Fix 3: Maintain Old Keys for Existing Members

Existing members must:

* Keep access to:

  * old group keys
  * new group keys

👉 So they can decrypt:

* Old messages → old key
* New messages → new key

---

### Fix 4: Key Distribution Strategy

When new member joins:

* Generate new group key
* Encrypt it for:

  * all current members (including new one)
* Do NOT send old keys to new member

---

### Fix 5: Debug Check

Log this on failure:

```ts
console.log("message.keyId", message.keyId);
console.log("availableKeys", userKeys);
```

👉 If key mismatch → that’s the bug

---

## Expected Behavior After Fix

### Existing Members

* Can still read ALL past messages ✅
* Can read new messages ✅

---

### New Member

* Cannot read old messages ✅
* Can read new messages after joining ✅

---

## Key Insight

This is not a UI issue.
This is not a state issue.

👉 This is a **cryptographic key management problem**.

---

## Final Note

To properly support E2EE group chats, you must implement:

* Key versioning
* Multi-key decryption support
* Proper key rotation handling

Once fixed:

* Your E2EE system will behave like real-world apps (Signal, WhatsApp)
* No message loss for existing users
* Correct privacy for new members
